1. Project Overview

We have a Python-based Streamlit application called Research Agent.

The application consists of:

app.py
search.py
summarize.py
agent.py
requirements.txt

The application uses:

Streamlit — Frontend/UI

Tavily — Web search

Groq — LLM-based answer generation

Python — Application runtime


The objective was to:

1. Containerize the application using Docker.


2. Store the Docker image in GitHub Container Registry.


3. Build the Docker image automatically using GitHub Actions.


4. Deploy the Docker image to Railway.


5. Make the application publicly accessible through a Railway domain.


6. Configure CI/CD so that future code changes automatically build a new Docker image and redeploy the application.

4. Dockerfile

We created a Dockerfile:

FROM python:3.12-slim

WORKDIR /app

COPY requirements.txt .

RUN pip install --no-cache-dir -r requirements.txt

COPY . .

EXPOSE 8501

CMD ["sh", "-c", "streamlit run app.py --server.address=0.0.0.0 --server.port=${PORT:-8501}"]

Let's understand each line.


---

4.1 Base Image

FROM python:3.12-slim

This uses Python 3.12 as the base environment.

slim is a smaller version of the Python image.


---

4.2 Working Directory

WORKDIR /app

Inside the container, the application will live in:

/app


---

4.3 Copy Dependencies

COPY requirements.txt .

This copies:

requirements.txt

into:

/app


---

4.4 Install Dependencies

RUN pip install --no-cache-dir -r requirements.txt

This installs all Python packages required by the application.

For example:

streamlit
tavily-python
groq
python-dotenv

and other dependencies.


---

4.5 Copy Application Code

COPY . .

This copies the project files into the container.

For example:

app.py
search.py
summarize.py
agent.py
requirements.txt


---

4.6 Expose Port

EXPOSE 8501

This documents the port used by the application.

However, Railway provides its own $PORT environment variable at runtime.


---

4.7 Start Streamlit

CMD ["sh", "-c", "streamlit run app.py --server.address=0.0.0.0 --server.port=${PORT:-8501}"]

This starts the Streamlit application.

The important part is:

--server.address=0.0.0.0

This allows the application to accept external connections.

The port is:

${PORT:-8501}

This means:

If PORT exists:
    use PORT

Otherwise:
    use 8501

Railway provides the $PORT variable to the container.

8. GitHub Actions Permissions

We added:

permissions:
  contents: read
  packages: write

This gives the workflow permission to:

contents: read

Read the repository code.

packages: write

Push Docker images to GHCR.

Without:

packages: write

GitHub Actions may fail when pushing the Docker image.

10. Login to GHCR

We use:

- name: Login to GitHub Container Registry
  uses: docker/login-action@v3
  with:
    registry: ghcr.io
    username: ${{ github.actor }}
    password: ${{ secrets.GITHUB_TOKEN }}

This authenticates Docker with GHCR.

The important part is:

GITHUB_TOKEN

GitHub automatically provides this token to GitHub Actions.

We don't manually create it.

11. Docker Image Name

We encountered an issue where the Docker image name had uppercase characters.

Docker image repository names must be lowercase.

The error was:

repository name must be lowercase

We fixed it using:

- name: Set Docker Image Name
  run: |
    echo "IMAGE_NAME=$(echo '${{ github.repository }}' | tr '[:upper:]' '[:lower:]')" >> $GITHUB_ENV

This converts the GitHub repository name to lowercase.

For example:

Adiking117/Research-Agent-Demo-CICD

becomes:

adiking117/research-agent-demo-cicd

Then the image becomes:

ghcr.io/adiking117/research-agent-demo-cicd:latest

17. Railway Port Configuration

Railway needs to know which port the application listens on.

We configured the Railway target port as:

8080

The container receives a Railway-provided $PORT.

The Docker startup command uses:

${PORT:-8501}

Therefore:

Railway
    │
    │ PORT=8080
    ▼
Docker Container
    │
    ▼
Streamlit
    │
    ▼
0.0.0.0:8080

The application needs to bind to:

0.0.0.0

rather than:

localhost

because Railway needs to access the application from outside the container.

19. Current CI/CD Pipeline

Our current GitHub Actions workflow is:

name: CI/CD Pipeline

on:
  push:
    branches:
      - main

permissions:
  contents: read
  packages: write

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Repository
        uses: actions/checkout@v4

      - name: Login to GitHub Container Registry
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Set Docker Image Name
        run: |
          echo "IMAGE_NAME=$(echo '${{ github.repository }}' | tr '[:upper:]' '[:lower:]')" >> $GITHUB_ENV

      - name: Build Docker Image
        run: |
          docker build \
            -t ghcr.io/${IMAGE_NAME}:latest \
            .

      - name: Push Docker Image
        run: |
          docker push \
            ghcr.io/${IMAGE_NAME}:latest

This currently provides:

Continuous Integration (CI)

because every push to main automatically:

1. Checks out the code.


2. Logs into GHCR.


3. Builds the Docker image.


4. Pushes the Docker image to GHCR.

The complete CD process should be:

    ▼

    ▼

    │
    ▼

    │
    ▼
Restart Container
    │
    ▼

The CD part is the remaining automation step we were working on.
