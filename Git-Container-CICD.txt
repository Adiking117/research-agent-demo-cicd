Research Agent — Containerization, CI/CD & Railway Deployment Documentation

1. Overview

This document explains how to containerize an existing Python application using Docker, create a CI pipeline using GitHub Actions, build and publish Docker images to GitHub Container Registry (GHCR), and deploy the application on Railway.

The overall deployment architecture is:

6. Docker Containerization

6.1 What is Docker?

Docker packages an application and its dependencies into a portable container.

Without Docker, the application depends on the environment where it is executed:

Operating System
       +
Python Version
       +
Installed Packages
       +
Application Code

With Docker, the environment is packaged consistently:

Docker Image
│
├── Base Python Image
├── Python Dependencies
└── Application Code

The Docker image can then be executed as a container:

Docker Image
      │
      ▼
Docker Container
      │
      ▼
Running Application


---

7. Creating the Dockerfile

Create a file named:

Dockerfile

A suitable Dockerfile for a Streamlit-based Python application is:

FROM python:3.12-slim

WORKDIR /app

COPY requirements.txt .

RUN pip install --no-cache-dir -r requirements.txt

COPY . .

EXPOSE 8501

CMD ["sh", "-c", "streamlit run app.py --server.address=0.0.0.0 --server.port=${PORT:-8501}"]


---

8. Understanding the Dockerfile

Base Image

FROM python:3.12-slim

This uses a lightweight Python 3.12 image.


---

Working Directory

WORKDIR /app

All application commands will execute inside:

/app


---

Copy Requirements

COPY requirements.txt .

The dependency file is copied into the container.


---

Install Dependencies

RUN pip install --no-cache-dir -r requirements.txt

Docker installs all Python dependencies required by the application.


---

Copy Application Code

COPY . .

The application source code is copied into the container.


---

Expose Port

EXPOSE 8501

Streamlit commonly runs on port 8501.


---

Start Application

CMD ["sh", "-c", "streamlit run app.py --server.address=0.0.0.0 --server.port=${PORT:-8501}"]

The application listens on:

0.0.0.0

This is important because the application needs to be accessible from outside the container.

The port logic is:

If PORT is provided:
    Use PORT

Otherwise:
    Use 8501

This makes the container suitable for both local Docker usage and Railway deployment.

13. GitHub Actions Workflow

A basic workflow is:

name: CI/CD Pipeline

on:
  push:
    branches:
      - main

  pull_request:
    branches:
      - main

permissions:
  contents: read
  packages: write

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Repository
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.12"

      - name: Install Dependencies
        run: |
          python -m pip install --upgrade pip
          pip install -r requirements.txt

      - name: Check Python Syntax
        run: |
          python -m py_compile app.py
          python -m py_compile agent.py
          python -m py_compile search.py
          python -m py_compile summarize.py

      - name: Log in to GitHub Container Registry
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Set Docker Image Name
        run: |
          echo "IMAGE_NAME=$(echo '${{ github.repository }}' | tr '[:upper:]' '[:lower:]')" >> $GITHUB_ENV

      - name: Build Docker Image
        run: |
          docker build \
            -t ghcr.io/${IMAGE_NAME}:latest \
            .

      - name: Push Docker Image to GHCR
        run: |
          docker push ghcr.io/${IMAGE_NAME}:latest


---

14. Understanding the CI Pipeline

Step 1 — Trigger

The workflow runs when:

Push to main

or:

Pull Request to main

occurs.


---

Step 2 — Checkout

uses: actions/checkout@v4

Downloads the repository code into the GitHub Actions runner.


---

Step 3 — Setup Python

uses: actions/setup-python@v5

Installs Python 3.12 for the CI environment.


---

Step 4 — Install Dependencies

pip install -r requirements.txt

This verifies that the application's dependencies can be installed successfully.


---

Step 5 — Validate Python Code

The pipeline runs:

python -m py_compile app.py

and similar commands for the other Python files.

This catches basic syntax errors.

If this step fails:

CI = FAILED

The pipeline stops.

16. Docker Image Naming Issue

During CI, the Docker build initially failed with:

repository name must be lowercase

The incorrect image name was:

ghcr.io/Adiking117/research-agent-demo-cicd:latest

The problem was the uppercase:

Adiking117

Docker image repository names must be lowercase.

The correct name is:

ghcr.io/adiking117/research-agent-demo-cicd:latest

The GitHub Actions workflow was updated to convert the repository name to lowercase:

IMAGE_NAME=$(echo "${{ github.repository }}" | tr '[:upper:]' '[:lower:]')

This generates:

ghcr.io/adiking117/research-agent-demo-cicd:latest

The Docker image can then be successfully pushed to GHCR.

23. Railway Port Configuration

Railway provides a dynamic PORT environment variable.

The Dockerfile uses:

--server.port=${PORT:-8501}

This means:

Railway
   │
   │ PORT = Railway assigned port
   ▼
Streamlit

If Railway doesn't provide a port, the application defaults to:

8501

The application also needs to listen on:

0.0.0.0

Therefore:

--server.address=0.0.0.0

is important.

Without this, the application may only be accessible from inside the container.

28. Current CI/CD Architecture vs Recommended Final Architecture

There is an important distinction.

Current setup

GitHub Actions builds and pushes an image:

GitHub
   │
   ▼
GitHub Actions
   │
   ▼
GHCR

At the same time, Railway can be connected directly to GitHub:

GitHub
   │
   ▼
Railway
   │
   ▼
Railway builds its own Docker image

This can result in Docker being built twice.

29. Recommended Production Architecture

For a cleaner CI/CD pipeline, the final architecture should be:

Developer
    │
    │ git push
    ▼
GitHub
    │
    ▼
GitHub Actions
    │
    ├── Run Tests
    │
    ├── Validate Code
    │
    ├── Build Docker Image
    │
    └── Push Image
            │
            ▼
           GHCR
            │
            ▼
         Railway
            │
            ▼
   Deploy Same Docker Image
            │
            ▼
       Live Application

This separates responsibilities:

Component	Responsibility

GitHub	Source code management
GitHub Actions	CI/CD automation
Docker	Application packaging
GHCR	Docker image storage
Railway	Application hosting
Railway Variables	Production secrets


The ideal pipeline is:

Code
 │
 ▼

 │
 ▼
GitHub Actions
 │
 ├── Test
 ├── Validate
 └── Build
      │
      ▼
    Docker
      │
      ▼
     GHCR
      │
      ▼

      │

 Production
